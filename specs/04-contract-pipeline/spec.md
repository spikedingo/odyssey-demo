# Spec 04 — Contract Generation Pipeline

## What

Establish the automated pipeline that derives frontend TypeScript types and React Query hooks from the backend schema — with zero manual duplication. The pipeline runs with a single command (`pnpm gen:contract`) and produces the contents of `packages/api-client`.

## Why

Manual type duplication between frontend and backend is one of the most common sources of subtle runtime bugs and maintenance overhead. By automating the full chain from Drizzle schema → OpenAPI spec → generated client, we guarantee that the frontend is always in sync with the backend, and we prove type safety across the HTTP boundary.

---

## Pipeline Overview

```
services/backend/src/db/schema.ts
  ↓  drizzle-zod
services/backend/src/db/validators.ts     (Zod schemas inferred from Drizzle)
  ↓  @hono/zod-openapi
services/backend/src/index.ts             (Hono app with OpenAPI routes registered)
  ↓  Wrangler build / tsx
GET /api/openapi.json                     (served at runtime; or written to file for Orval)
  ↓  Orval
packages/api-client/src/
  ├── generated/
  │   ├── endpoints/    (one file per tag/resource: orders.ts, customers.ts, …)
  │   └── model/        (TypeScript interfaces for all request/response shapes)
  └── index.ts          (re-exports everything)
```

## Package Roles

### `packages/types`

Holds **non-generated** shared types that are needed before generation runs, or that are not derivable from the OpenAPI spec:

```ts
// packages/types/src/index.ts
export type OrderStatus =
  | 'pending'
  | 'accepted'
  | 'preparing'
  | 'ready'
  | 'out_for_delivery'
  | 'completed'
  | 'cancelled';

export const ORDER_STATUS_LABELS: Record<OrderStatus, string> = {
  pending:          'Pending',
  accepted:         'Accepted',
  preparing:        'Preparing',
  ready:            'Ready',
  out_for_delivery: 'Out for Delivery',
  completed:        'Completed',
  cancelled:        'Cancelled',
};

export const VALID_TRANSITIONS: Record<OrderStatus, OrderStatus[]> = {
  pending:          ['accepted', 'cancelled'],
  accepted:         ['preparing', 'cancelled'],
  preparing:        ['ready'],
  ready:            ['out_for_delivery', 'completed'],
  out_for_delivery: ['completed'],
  completed:        [],
  cancelled:        [],
};
```

> The `VALID_TRANSITIONS` constant in `packages/types` is the display/UI reference. The backend's `orderService.ts` re-declares its own for type-safety at the service layer. Both must stay in sync — if state machine changes, update both.

#### Why `OrderStatus` lives in `packages/types` (not only in generated code)

The assignment says to avoid duplicated enums across frontend and backend. We deliberately keep a **small, hand-maintained** copy in `packages/types` for these reasons:

| Need | Why generated types alone are insufficient |
|---|---|
| UI labels (`ORDER_STATUS_LABELS`) | Display strings are not in the OpenAPI schema |
| Available actions (`VALID_TRANSITIONS`) | Business rules for which buttons to show — frontend needs this before/during render without an extra API call |
| Pre-generation bootstrap | Seed scripts, tests, and backend service layer reference status values before `pnpm gen:contract` runs |

**Sync rule**: `OrderStatus` union in `packages/types` must match the Drizzle `pgEnum('order_status', [...])` values exactly. When adding/removing a status: (1) update Drizzle schema, (2) update `packages/types`, (3) update backend `VALID_TRANSITIONS`, (4) run `pnpm gen:contract`. Generated Orval types for API responses remain the source of truth for **wire shapes**; `packages/types` is the source of truth for **UI/business-rule constants**.

This is **not** a handwritten frontend DTO — it is shared domain constants that OpenAPI cannot express.

### `packages/api-client`

Generated by Orval from `GET /api/openapi.json`. Contains:
- **Model types**: TypeScript interfaces for every schema registered in OpenAPI.
- **API hooks**: `useGetOrders`, `useCreateOrder`, `useUpdateOrderStatus`, etc. — all typed, using `@tanstack/react-query`.
- **Axios/fetch client**: Orval can generate a custom axios instance or a fetch wrapper. We use **fetch** to avoid an extra dependency.

The `src/` directory is gitignored and regenerated by `pnpm gen:contract`.

## Orval Configuration

`packages/api-client/orval.config.ts`:

```ts
import { defineConfig } from 'orval';

export default defineConfig({
  odysseyApi: {
    input: {
      target: process.env.OPENAPI_URL ?? 'http://localhost:8787/api/openapi.json',
    },
    output: {
      mode: 'tags-split',            // one file per OpenAPI tag
      target: './src/generated',
      client: 'react-query',
      httpClient: 'fetch',
      baseUrl: process.env.EXPO_PUBLIC_API_BASE_URL ?? 'http://localhost:8787',
      override: {
        mutator: {
          path: './src/customFetch.ts',
          name: 'customFetch',
        },
        query: {
          useQuery: true,
          useMutation: true,
        },
      },
    },
  },
});
```

`packages/api-client/src/customFetch.ts` — the custom fetcher that reads `EXPO_PUBLIC_API_BASE_URL` and adds any common headers (e.g., `Content-Type: application/json`).

## `gen:contract` Script Flow

The root `pnpm gen:contract` command does:

1. **Build backend** — `turbo run build --filter=backend` compiles the Hono app.
2. **Start backend** (or use static file) — Either spin up backend momentarily to serve `/api/openapi.json`, OR use the `@hono/zod-openapi` static export to write the spec to a file without running the server:
   ```bash
   tsx services/backend/src/openapi.ts --output ./openapi.json
   ```
   Prefer the static file approach for reliability in CI.
3. **Run Orval** — `pnpm --filter=api-client exec orval` reads the spec file and regenerates `src/generated/`.
4. **Type-check api-client** — `pnpm --filter=api-client typecheck` validates generated output.

Full script in root `package.json`:
```json
"gen:contract": "pnpm --filter=backend run build:openapi && pnpm --filter=api-client exec orval && pnpm --filter=api-client typecheck"
```

`build:openapi` in `services/backend/package.json`:
```json
"build:openapi": "tsx src/openapi-export.ts"
```

`src/openapi-export.ts` — instantiates the Hono app, calls `.getOpenAPIDocument()` or fetches `/api/openapi.json` from the built app, writes to `../../openapi.json`.

## OpenAPI Tagging Convention

All Hono routes must declare an `tags` field to enable Orval's `tags-split` mode:

| Tag | Routes |
|---|---|
| `health` | `GET /health` |
| `categories` | `GET/POST/DELETE /categories` |
| `menu-items` | `GET/POST/PATCH/DELETE /menu-items` |
| `customers` | `GET/POST /customers`, `GET/PATCH /customers/:id` |
| `orders` | `GET/POST /orders`, `GET /orders/:id`, `PATCH /orders/:id/status` |
| `settings` | `GET/PATCH /settings` |
| `home` | `GET /home/summary` |

Orval generates one file per tag in `src/generated/endpoints/`.

---

## Generated Hook Naming Convention

Orval generates hooks with the pattern `use{Method}{OperationId}`. With the `operationId` values we define in spec 02:

| operationId | Generated hook |
|---|---|
| `listOrders` | `useListOrders` |
| `createOrder` | `useCreateOrder` (mutation) |
| `getOrder` | `useGetOrder` |
| `updateOrderStatus` | `useUpdateOrderStatus` (mutation) |
| `listCustomers` | `useListCustomers` |
| `getCustomer` | `useGetCustomer` |
| `listMenuItems` | `useListMenuItems` |
| `getHomeSummary` | `useGetHomeSummary` |
| `getSettings` | `useGetSettings` |
| `updateSettings` | `useUpdateSettings` (mutation) |

## `packages/api-client` Structure (post-generation)

```
packages/api-client/
├── src/
│   ├── customFetch.ts          # hand-written; reads EXPO_PUBLIC_API_BASE_URL
│   ├── generated/
│   │   ├── endpoints/
│   │   │   ├── orders.ts       # useListOrders, useCreateOrder, useGetOrder, useUpdateOrderStatus
│   │   │   ├── customers.ts
│   │   │   ├── menuItems.ts
│   │   │   ├── categories.ts
│   │   │   ├── settings.ts
│   │   │   └── home.ts
│   │   └── model/
│   │       ├── order.ts        # Order, OrderDetail, OrderSummary, CreateOrderBody, ...
│   │       ├── customer.ts
│   │       ├── menuItem.ts
│   │       └── ...
│   └── index.ts                # re-exports from generated/ + customFetch
├── orval.config.ts
├── package.json
└── tsconfig.json
```

Files under `src/generated/` are gitignored. `src/customFetch.ts` is committed (hand-written, stable).

## Acceptance Criteria

- [ ] Running `pnpm gen:contract` from the workspace root completes without error.
- [ ] `packages/api-client/src/generated/` is populated after running `gen:contract`.
- [ ] `packages/api-client/src/generated/model/order.ts` exports a TypeScript type with `status` typed as the 7-value union (not plain `string`).
- [ ] `packages/api-client/src/generated/endpoints/orders.ts` exports `useListOrders`, `useCreateOrder`, `useGetOrder`, `useUpdateOrderStatus`.
- [ ] `pnpm --filter=api-client typecheck` passes after generation.
- [ ] `apps/dashboard` imports types only from `packages/api-client` or `packages/types` — no imports from `services/backend`.
- [ ] No file in `apps/dashboard/src/` manually declares a type that mirrors an API response shape.
- [ ] `openapi.json` at workspace root validates as valid OpenAPI 3.0.
- [ ] Running `gen:contract` twice produces identical output (idempotent).
- [ ] The `src/generated/` directory does not appear in `git status` (gitignored).
